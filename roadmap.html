User Testing

The User Test for Roadmaps' business travel oriented software

Summary

Research

During the research phase I searched for the best way to understand what part of the current system needed to be tested. By speaking with the employees of Roadmap and getting the focus of the timeline part of the app that needs to be extensively tested, research questions were made. The main research question was:

“Which measures can be taken to improve the usability of the Roadmap application timeline.”

The conversations with the employees of Roadmap made clear they had no clear idea about their own usability since they rarely do any user tests. The companies they provide the application for doesn’t want to let its employees get involved in User Testing. Both of these things factor in their inability to test the product themselves. From the main research question smaller research questions were comprise based off the heuristics done on the current timeline and app. These also each got their own concrete measurement questions which will directly get answered in the user test.

  1. How clear does the user think the timeline is
    • Can the user book an Uber from the timeline?
    • How many steps does the user take to achieve booking an Uber?
  2. What does the user think about the navigation within the software?
    • How does the user experience searching for a restaurant in the vicinity?
    • How long does it take the user to find the closest restaurants?
    • How many steps does it take for the user to find a restaurant?
  3. What does the user think about the tabs with extra information in the timeline?
    • Does the user notice that the appointments from the agenda communicate with the other aspects of the timeline?
    • How does the user experience the communication between these elements on the timeline?

After setting up the main research question and all the smaller questions from that point on to break the test down in smaller chunks the test was set up with scenario’s and test tasks. The 5 scenarios were set up with no more than 2 tasks each. They involve using only the timeline and are focused on answering the research questions.

The Test

There were 5 candidates for the user test, as 5 is the recommended number by J. Nielsen who states that 5 is the ideal number to test to find most problems without being redundant. The test itself was positioned as seen below. Both the points listed for improvement as the positive feedback were given space to be listed throughout the test. This is visible for example in the post-test questionnaire where points for improvement are asked together with the question to list a positive note about the application. This way the test focusses on all the aspects, and not just what doesn’t work. It also really helps to include some positive feedback for the customers.

Iterations

During the first test a few question phrasings were incorrect and adjusted accordingly. Based on the pacing of this first test the other tests would also be measured. This means that the tests itself had a proper time set per scenario and task, and the task wasn’t poorly phrased or too obscure.

The result

The result consists of a rapport with key highlights per scenario that were important and found in most/all user tests. It also houses a list of recommendations off of these findings. The findings are compartmentalized into a “could be adjusted” group and a “functions that could be added” group. With the former focusing on the most important aspects to improve and the latter on what could make the whole experience a better product altogether. All these recommendations are based and backed up by the user tests.

The recommendations are then put into a visual form to help with the presentability and to get a better picture of the improvements. These are done per key recommendation and are edited within the visual layout of the app itself.

Presentation

The presentation was held at the main office of Roadmap in Delft, where 2 key members of the organization would listen to the results as they were presented. The visuals from the recommendations were shown with the rest of the rapport backing up all the info and suggestions that were given.

Conclusion

The test itself was successful, a lot of useful information was given and processed accordingly. The information with the rapport can help Roadmap make their own adjustments and base them off data that is tested and accounted for. By making recommendations on what parts to focus on and improve without telling them how to improve it the role of a tester was fully embraced in this project.
